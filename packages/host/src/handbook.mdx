# Handbook

Living front-end guidelines for the squid·ware UI monorepo (`@quido-ui/*` packages). Focus is technical practice, not visual design. Covers React, Vue, and Angular.

## Code quality

### Style and lint

ESLint, Prettier, and TypeScript enforce style. Install the matching editor extensions. Prefer the shared ESLint setup for the stack you are in (framework rules, TypeScript, Storybook, a11y where configured, Prettier).

### Commits

Use [Conventional Commits](https://www.conventionalcommits.org/):

- `fix(auth): correct login button padding. GH-100`
- `feat(auth): add GitHub login. GH-100`

Say why the change exists when it helps. See [Contribution](?path=/docs/contribution--docs) for the full PR flow.

### Git flow

- One feature branch per ticket, for example `feature/GH-100-issue`.
- Branch from `main`, open a PR into `main`, delete the branch after merge.

## Shared conventions

### Naming

- Component folders and files: kebab-case, matching the component name.
- Stories: `[component-name].stories.tsx` (or `.ts` / Vue equivalent) next to the component.
- Unit tests: `[name].spec.tsx` (or package equivalent).
- Cypress component tests (when used): `[component-name].cy.tsx`.

### Placement in this repo

- Shared UI: `packages/<framework>/src/components/`
- Feature demos (React): `packages/react/src/features/<feature>/components/`
- Design tokens and icon stories (Angular): `packages/angular/src/system/`

### Exports

Prefer named exports over `export default` so names stay stable. Avoid barrel files that hurt tree-shaking.

### Test ids

Use `data-testid` with a BEM-like pattern:

- Single instance: `card`, `card__title`
- Many instances: `list__item-${id}`

## Styling in quido-ui

Native CSS (or CSS-in-JS that embeds in the component file). Styles live with the component:

| Framework | Approach |
| --------- | -------- |
| React | styled-components + CVA in the same `.tsx`. Tokens from `packages/react/src/index.css`. No CSS Modules or Tailwind utilities for package components. |
| Vue | `<style scoped>` in the SFC. Tokens from `packages/vue/src/assets/base.css`. |
| Angular | Component `styles` / `styleUrls` with encapsulation. Tokens from `packages/angular/src/styles.css`. |

Variants use CVA where the package already does (`q-*`, `variant--*`, `size--*`). Align variant names with Figma when designs exist.

Do not pull in a full third-party UI kit (for example Material) for these packages. Opt-out source you can edit is the point. For behaviour without visuals, prefer headless libs already in use: Radix (React), Reka UI (Vue), Angular CDK (Angular).

## React notes

Starters (outside this repo, for apps):

- [Next.js Starter](https://github.com/qhoekman/next-starter)
- [Next.js Monorepo](https://github.com/qhoekman/next-monorepo)

Stack often used with React apps built here: TypeScript, Vite or Next.js, Testing Library, Storybook, Chromatic for visual diffs.

State: React Query (or similar) for server state; Redux Toolkit or Zustand for client state when you need a store. Keep state immutable and serializable when using Redux.

Forms and tables: React Hook Form, TanStack Table when needed. Validate with Zod when schemas help.

## Angular notes

Stack: Angular + TypeScript, Storybook, Testing Library / Jest, Cypress when configured, Chromatic for visual diffs.

Behaviour: Angular CDK. Prefer directive-host components (`button[qui-button]`) and `<ng-content>` so consumers keep host attributes. See [Technical Design](?path=/docs/technical-design--docs).

Client state in larger apps: NgRx when a store is justified. Pair start / success / error actions for async flows.

## Vue notes

Stack: Vue 3 + TypeScript, Vite, Storybook, Vitest for unit tests when present.

Behaviour: Reka UI for headless patterns. Keep components as SFCs with scoped CSS and CVA variants, matching React/Angular class naming where possible.

## Assets

- Put static files in `public` (or the package equivalent).
- Export SVGs from Figma; optimize with [SVGOMG](https://jakearchibald.github.io/svgomg/) when needed.
- Icons: SVG as components, use `currentColor`, avoid fixed width/height so size can scale.
- Prefer responsive images (`picture` / `srcset`, or Next.js `Image` in Next apps).

## Testing

| Layer | Tool (typical) | What to cover |
| ----- | -------------- | ------------- |
| Unit | Jest / Vitest | Pure functions, hooks, reducers, services |
| Component | Testing Library and/or Cypress CT | Render, interactions, edge data |
| E2E | Cypress | Happy path, failure path, roles |
| Visual | Chromatic | Storybook story diffs |

Prefer Testing Library queries that mirror how people use the UI. Add axe (for example Cypress Axe) for automated a11y checks where Cypress is in place.

## Accessibility

Follow [WCAG 2.1](https://www.w3.org/TR/WCAG21/) and [WAI-ARIA](https://www.w3.org/TR/wai-aria/). Use semantic HTML, meaningful `alt` / labels, and keyboard support. Validate in Storybook with browser tools and manual checks; `@storybook/addon-a11y` can be added per package if you want it in the UI.

## Documentation

- Repo docs for this design system live in host Storybook (this site).
- Component stories use CSF in TypeScript next to the component.
- App repos should still keep README, CONTRIBUTING, architecture, and security docs as needed.

## Security

Dependency scanning with Snyk (or equivalent CI). Sanitize untrusted HTML with DOMPurify before `innerHTML` / `dangerouslySetInnerHTML` / MDX that embeds HTML.

## Package management

This monorepo uses **pnpm** workspaces and **Turborepo**. Script names in `package.json`: kebab-case, often with `:` for groups. Version published packages with [semver](https://semver.org/) when you publish.

## CI

Pipelines should install with a frozen lockfile, then build, unit test, component test, e2e, and visual checks as the package supports them.
